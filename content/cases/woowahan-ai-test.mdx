## 왜 이 사례가 중요한가

오래 운영된 서비스일수록 **레거시 코드**가 많을 수밖에 없습니다.
빠른 비즈니스 요구를 따라가다 보면 테스트 코드는 뒷전이 되고, 엣지케이스나 예외 상황에 대한 검증이 부족해집니다.
이런 상황에서 **AI를 활용해 테스트 코드를 작성하는 것**은 레거시 코드 개선의 현실적인 첫걸음이 될 수 있습니다.

우아한형제들 커머스팀은 7개 레포지토리를 관리하면서 테스트 커버리지 부족 문제를 AI로 해결하려 했고, 여러 번의 시행착오 끝에 **실용적인 방법론**을 찾아냈습니다.

## 한눈에 보는 결과

| 항목 | AI만 사용 | 플러그인 + AI |
|------|-----------|---------------|

| **컴파일 성공률** | 85% | 95% |
| **메서드 커버리지** | 67% | 95% |
| **100개 클래스 기준** | 16.7시간 | 5시간 (11.7시간 절약) |

## 여정: 세 번의 시도

### 1단계 — AI 에이전트만으로 시도

ChatGPT, Claude, Gemini 같은 AI 도구를 비교한 뒤, IntelliJ에 통합되는 **Amazon Q**를 선택했습니다.

| 도구 | IDE 환경 | 의존성 인식 | 컨텍스트 |
|------|----------|------------|---------|
| GitHub Copilot | VS Code 기반 | 제한적 | 파일 단위 |
| Amazon Q | IntelliJ 통합 | 우수 | 프로젝트 전체 |
| Cursor | 독립 IDE | 우수 | 프로젝트 전체 |

기존 IntelliJ 환경을 유지할 수 있다는 점이 Amazon Q를 선택한 가장 큰 이유였습니다.

하지만 10개 클래스에 테스트를 작성하면서 **반복되는 문제**가 나타났습니다:

- 매번 같은 팀 컨벤션 설명을 반복해야 함
- 클래스마다 평균 10분 소요
- 생성된 코드의 약 15%에서 빌드 오류 발생

### 2단계 — IntelliJ 플러그인으로 자동화 시도

반복 작업을 자동화하기 위해 **IntelliJ 플러그인**을 개발했습니다.
Gemini API를 호출해 완성된 테스트 코드를 자동 생성하는 방식이었습니다.

**결과는 참담했습니다:**

| 결과 | 건수 | 비율 |
|------|------|------|
| 테스트 생성 성공 | 10/10 | 100% |
| 컴파일 성공 | 1/10 | **10%** |
| 수정 없이 바로 실행 | 0/10 | **0%** |

**발견된 주요 문제:**
- **기존 테스트 손실** — 테스트를 "추가"하려 했더니 기존 테스트가 완전히 덮어쓰기 됨
- **AI 할루시네이션** — 존재하지 않는 클래스, 잘못된 import, 타입 불일치
- **멀티모듈 환경의 복잡성** — 같은 이름의 클래스가 여러 모듈에 존재할 때 AI가 혼란

<Callout type="warning">
<strong>핵심 교훈</strong>: AI가 생성한 코드는 "그럴듯하지만" 실제로는 동작하지 않는 경우가 많았습니다. 특히 실제 운영 중인 멀티모듈 프로젝트에서 이 문제가 심했습니다.
</Callout>

자동 수정(재시도) 방식도 실험했지만, 3회 시도 후에도 35%는 실패했고 시간·비용만 증가했습니다.

### 3단계 — 템플릿 + Amazon Q (최종 해법)

근본 원인을 분석한 결과, **역할 분담**이 핵심이었습니다:

```
기존: 플러그인 → AI → 완성된 테스트 (오류 가능)
개선: 플러그인 → 컴파일 보장 템플릿 → Amazon Q → 완성된 테스트
```

**플러그인이 하는 일** (AI가 어려워하는 부분):
- 코드 분석으로 **모든 분기를 빠짐없이** 테스트 케이스로 추출
- **컴파일이 보장되는** 템플릿(뼈대) 생성
- 기존 테스트 보존, 팀 컨벤션 자동 적용

**Amazon Q가 하는 일** (AI가 잘하는 부분):
- 템플릿의 `TODO` 부분에 구체적인 테스트 로직 구현
- IDE 컨텍스트를 활용한 정확한 코드 생성

실제 복잡한 레거시 코드(5개 의존성, 중첩 조건문)로 비교한 결과:

| 항목 | Amazon Q만 | 플러그인 + Amazon Q |
|------|-----------|-------------------|
| 생성된 테스트 수 | 4개 | **12개** |
| 커버리지 | 60% | **95%** |
| 누락 엣지케이스 | 5개 | **0개** |
| 소요 시간 | 8분 | **4분** |

## 핵심 인사이트

이 사례에서 얻을 수 있는 교훈을 정리하면:

### 1. 완벽한 자동화는 환상이다

AI는 "**보조 도구**"이지 "대체 도구"가 아닙니다.
사람의 검증과 판단이 반드시 필요하며, AI에게 맡길 부분과 사람이 해야 할 부분을 명확히 나누는 것이 핵심입니다.

### 2. AI가 잘하는 것과 못하는 것을 구분하라

- **AI가 잘하는 것**: 패턴이 정해진 코드 구현, IDE 컨텍스트 기반 코드 완성
- **AI가 못하는 것**: 모든 엣지케이스 파악, 컴파일 보장, 멀티모듈 환경의 정확한 import 해결

이 구분에 따라 **도구 간 역할을 나누는 것**이 실용적인 접근입니다.

### 3. 팀과 함께 개선하라

혼자 도구를 만들어 배포하는 것보다, 팀과 함께 사용하며 피드백을 받고 개선하는 과정이 중요합니다.
우아한형제들 팀도 첫 버전의 문제점을 팀원들과 함께 발견하고, 그 피드백으로 최종 방식을 만들어냈습니다.

### 4. 레거시 코드 개선의 현실적 시작점

테스트 코드 작성은 레거시 코드 개선의 **가장 안전한 첫걸음**입니다.
기존 코드를 건드리지 않으면서 안전망을 만들 수 있고, AI가 도와줄 수 있는 영역이기도 합니다.
