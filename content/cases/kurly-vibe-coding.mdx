## 왜 이 사례가 중요한가

바이브코딩이 유행이지만, 실무에서 "프롬프트 한 줄로 뚝딱"은 환상입니다.
스펙을 통째로 넣으면 중간 내용이 빠지고, 대화가 길어지면 컨벤션을 잊고, 한 번의 오해가 눈덩이처럼 커집니다.

컬리의 박재영 개발자는 이러한 문제들이 <strong>LLM의 구조적 한계</strong>(Lost in the Middle, 작업 기억 부재, 학습 데이터 편향)에서 비롯됨을 분석하고, Claude Code의 도구들로 이를 <strong>시스템적으로 보완하는 8가지 전략</strong>을 정리했습니다.

---

## LLM의 인지적 한계 — 왜 문제가 발생하는가

| 한계 | 설명 | 증상 |
|------|------|------|
| **Lost in the Middle** | 긴 컨텍스트에서 중간 정보가 누락됨 | 스펙의 6~25페이지 내용이 빠짐 |
| **학습 데이터 편향** | 과거에 더 많았던 패턴으로 회귀 | Zustand 지시했는데 Redux로 돌아감 |
| **오류 중첩** | 초반 오해가 이후 모든 코드에 전파 | 요청 안 한 2FA가 시스템 전체에 녹아듦 |
| **작업 기억 부재** | 여러 규칙을 동시에 추적하지 못함 | 5개 규칙 중 4, 5번이 빠짐 |

<Callout type="warning">
<strong>핵심 발견</strong>: LLM은 인간이 만든 데이터로 학습했기 때문에 인간 인지의 편향을 물려받았지만, 인간이 한계를 극복하는 전략(쪼개고, 반복하고, 점검하기)은 스스로 실행하지 못합니다.
</Callout>

## 8가지 해결 전략

### 인간의 전략을 LLM에 적용하는 구조

| 인간의 전략 | LLM의 한계 | Claude Code 도구 |
|------------|-----------|----------------|
| 작게 쪼개서 처리 | Lost in the Middle | 멀티턴 대화 |
| 실행 전 계획 점검 | 오류 중첩 | Plan 모드 |
| 체크리스트로 관리 | 여러 항목 동시 추적 실패 | Todo |
| 역할 분담 | 단일 컨텍스트 한계 | 서브에이전트 |
| 깊이 생각하기 | 복잡한 추론 필요 | Extended Thinking |
| 중요사항 주기적 확인 | 초기 정보 희석 | CLAUDE.md |
| 반복 검증 자동화 | 수동 확인 한계 | Agent Skills |
| 불필요한 정보 정리 | 컨텍스트 비활성화 불가 | Compact |

### 방법 1: 작은 요청의 멀티턴 대화

```
1턴: "로그인 화면 만들어줘. 이메일과 비밀번호로 로그인하는 기본 형태야."
→ 결과 확인 ✓
2턴: "좋아. 로그인 성공하면 /dashboard로 이동하게 해줘."
→ 결과 확인 ✓
3턴: "비밀번호 3회 틀리면 계정 30분 잠금해줘."
→ 결과 확인 ✓
```

각 턴이 짧아서 정보 누락이 없고, 명확해서 오해가 없으며, 매번 확인하므로 오류가 누적되지 않습니다.

### 방법 2: Plan 모드 — 실행 전 방향 검증

Plan 모드는 에이전트가 코드를 수정하기 전에 **먼저 계획을 수립**하도록 강제합니다.

| Mode | 설명 | 용도 |
|------|------|------|
| Default | 매번 권한 요청 | 신중한 작업 |
| Auto-accept | 파일 수정 자동 승인 | 반복 작업 |
| Plan | 계획 수립 후 승인 | 복잡한 작업, 방향 검증 |

### 방법 3: Todo — 체크리스트로 누락 방지

10개 파일을 한 번에 수정하면 중간에 빠지는 파일이 생깁니다. Todo로 목록을 먼저 만들고 하나씩 처리하면 누락 없이 작업을 완료할 수 있습니다.

### 방법 4: 서브에이전트 — 컨텍스트 분리

전체 코드베이스를 한 번에 분석하는 대신, **도메인별로 독립 에이전트를 실행**해 각자 작은 컨텍스트에서 정밀하게 작업합니다.

### 방법 5: Extended Thinking — 깊이 있는 추론

**"왜?"라는 질문이 필요한 작업**(디버깅, 레거시 분석, 아키텍처 결정)에는 Extended Thinking이 효과적이고, "무엇을"만 명확한 작업(CRUD, 보일러플레이트)에는 기본 모드가 더 효율적입니다.

### 방법 6: CLAUDE.md — 계층적 컨텍스트 관리

| 위치 | 용도 | 로드 시점 |
|------|------|----------|
| root/CLAUDE.md | 프로젝트 온보딩 (구조, 명령어, 흐름) | 항상 |
| src/xxx/CLAUDE.md | 해당 디렉토리 특화 컨벤션 | 디렉토리 접근 시 |

### 방법 7: Agent Skills — 검사와 워크플로우 자동화

단순히 룰을 나열하는 게 아니라, **실제로 검사하고 검증하는 로직**을 Skill로 만들어 반복 위반을 자동으로 잡아냅니다.

### 방법 8: Compact — 컨텍스트 정리

대화가 10턴 이상 지속되거나, 새로운 작업으로 전환할 때 `/compact`로 컨텍스트를 요약해 집중력을 유지합니다.

## 핵심 인사이트

### 1. "작게 나누고, 자주 확인하고, 반복은 스킬로 만들어라"

이 한 줄이 바이브코딩의 핵심입니다. LLM의 한계를 이해하고, 그 한계를 도구로 보완하는 것이 **예측 가능한** 결과를 만듭니다.

### 2. LLM의 한계를 알면 전략이 보인다

문제의 원인을 모르면 해결책도 모릅니다. Lost in the Middle, 학습 데이터 편향, 작업 기억 한계 — 이 세 가지를 이해하면 왜 특정 도구가 필요한지 자연스럽게 알 수 있습니다.

### 3. 프롬프트의 품질이 도구의 효과를 결정한다

아무리 좋은 도구를 써도, 요구사항 자체가 모호하면 누락이 발생합니다.
**"누가 읽어도 같은 것을 상상할 수 있어야 한다"**는 원칙으로 프롬프트를 작성하세요.
